## 为什么需要微服务

### 单体架构的痛点

```
┌─────────────────────────────────────┐
│         单体应用 (Monolith)          │
├─────────────────────────────────────┤
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐   │
│  │用户 │ │订单 │ │支付 │ │库存 │   │
│  │模块 │ │模块 │ │模块 │ │模块 │   │
│  └─────┘ └─────┘ └─────┘ └─────┘   │
│                                     │
│  ┌─────────────────────────────┐   │
│  │       共享数据库             │   │
│  └─────────────────────────────┘   │
└─────────────────────────────────────┘
```

**典型问题：**

| 问题 | 影响 | 严重程度 |
|------|------|----------|
| 代码耦合 | 修改一处，影响全局 | 🔴 高 |
| 部署困难 | 全量部署，风险集中 | 🔴 高 |
| 扩展性差 | 只能整体扩展 | 🟡 中 |
| 技术栈锁定 | 难以引入新技术 | 🟡 中 |
| 团队效率 | 多人协作冲突 | 🔴 高 |

### 微服务的优势

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│  用户服务 │  │  订单服务 │  │  支付服务 │
│  ┌────┐  │  │  ┌────┐  │  │  ┌────┐  │
│  │ API│  │  │  │ API│  │  │  │ API│  │
│  └────┘  │  │  └────┘  │  │  └────┘  │
│  ┌────┐  │  │  ┌────┐  │  │  ┌────┐  │
│  │ DB │  │  │  │ DB │  │  │  │ DB │  │
│  └────┘  │  │  └────┘  │  │  └────┘  │
└──────────┘  └──────────┘  └──────────┘
     ↓              ↓              ↓
  独立部署       独立扩展       独立技术栈
```

**核心收益：**

- ✅ **独立部署** - 单个服务更新不影响其他服务
- ✅ **弹性扩展** - 按需求扩展热点服务
- ✅ **技术多样** - 不同服务使用最适合的技术栈
- ✅ **故障隔离** - 单点故障不影响全局
- ✅ **团队自治** - 小团队全权负责一个服务

class EventPublisher:

## 架构设计原则

### SOLID 原则在微服务中的应用

#### 1. 单一职责原则 (SRP)

```
✅ 正确：每个服务只做一件事
- 用户服务：只处理用户相关逻辑
- 订单服务：只处理订单相关逻辑
- 支付服务：只处理支付相关逻辑

❌ 错误：服务职责混乱
- 电商服务：处理用户 + 订单 + 支付 + 库存
```

#### 2. 开闭原则 (OCP)

```python
# 通过事件扩展功能，不修改原有服务

# 订单服务只负责创建订单
class OrderService:
    def create_order(self, order_data):
        order = self._save_order(order_data)
        # 发布事件，其他服务可以订阅
        self._publish_event("ORDER_CREATED", order)
        return order

# 库存服务订阅事件，独立处理库存扣减
class InventoryListener:
    @subscribe("ORDER_CREATED")
    def on_order_created(self, event):
        self._deduct_inventory(event.items)
        
# 邮件服务订阅事件，独立发送通知
class EmailListener:
    @subscribe("ORDER_CREATED")
    def on_order_created(self, event):
        self._send_confirmation_email(event.user_id)
```

### CAP 定理的权衡

```
┌──────────────────────────────────────────┐
│              CAP 定理                     │
│                                          │
│  分布式系统最多同时满足以下两项：           │
│  - Consistency (一致性)                   │
│  - Availability (可用性)                  │
│  - Partition tolerance (分区容错性)       │
│                                          │
│  微服务架构选择：AP + 最终一致性            │
└──────────────────────────────────────────┘
```

**实践建议：**

| 场景 | 选择 | 理由 |
|------|------|------|
| 用户登录 | CP | 必须保证数据一致 |
| 商品浏览 | AP | 可以接受短暂不一致 |
| 订单创建 | CP | 涉及资金，必须一致 |
| 评论点赞 | AP | 可以异步同步 |

## 实战案例

### 案例：电商系统微服务化

#### 改造前（单体架构）

```python
# monolithic_app.py
class EcommerceApp:
    def __init__(self):
        self.db = Database()
    
    def place_order(self, user_id, items):
        # 用户验证
        user = self.db.query("SELECT * FROM users WHERE id=?", user_id)
        
        # 库存检查
        for item in items:
            stock = self.db.query(
                "SELECT stock FROM products WHERE id=?", 
                item['id']
            )
            if stock < item['quantity']:
                raise Exception("库存不足")
        
        # 创建订单
        order_id = self.db.insert(
            "INSERT INTO orders (user_id, total) VALUES (?, ?)",
            user_id, sum(i['price'] * i['qty'] for i in items)
        )
        
        # 扣减库存
        for item in items:
            self.db.execute(
                "UPDATE products SET stock=stock-? WHERE id=?",
                item['quantity'], item['id']
            )
        
        # 发送通知
        self.send_email(user.email, "订单确认")
        
        return order_id
```

## 常见陷阱

### 🚨 陷阱 1：过度拆分

```
❌ 错误示范
- UserService (仅 200 行代码)
- UserAuthMicroService (仅 150 行代码)
- UserProfileNanoService (仅 100 行代码)

✅ 正确做法
- UserContext (包含用户 + 认证 + 资料，约 2000 行代码)
```

**判断标准：**
- 服务间调用次数 > 服务内调用次数 → 拆分过细
- 部署一个功能需要更新 5+ 服务 → 拆分过细
- 团队人数 < 服务数量 → 拆分过细

### 🚨 陷阱 2：分布式单体

```
❌ 错误示范
- 服务独立部署，但共享数据库
- 服务间强耦合，必须同时部署
- 所有服务使用相同技术栈和版本

✅ 正确做法
- 每个服务独立数据库
- 服务间通过 API/事件通信
- 允许不同服务使用不同技术栈
```

### 🚨 陷阱 3：忽视数据一致性

```python
# ❌ 错误：忽略分布式事务
async def create_order(order_data):
    await user_service.verify(order_data.user_id)
    await inventory_service.deduct(order_data.items)
    await order_service.create(order_data)
    # 如果最后一步失败，前面无法回滚！

# ✅ 正确：使用 Saga 模式
async def create_order_with_saga(order_data):
    saga = Saga()
    try:
        await saga.add_step(
            action=lambda: user_service.verify(order_data.user_id),
            compensation=lambda: None  # 验证无需回滚
        )
        await saga.add_step(
            action=lambda: inventory_service.lock(order_data.items),
            compensation=lambda: inventory_service.release(order_data.items)
        )
        await saga.add_step(
            action=lambda: order_service.create(order_data),
            compensation=lambda: order_service.cancel(order_data.id)
        )
        await saga.execute()
    except Exception as e:
        await saga.rollback()
        raise e
```

## 总结

### 微服务决策树

```
开始
  │
  ↓
团队规模 > 10 人？
  ├─ 否 → 继续使用单体
  └─ 是 → 继续
       │
       ↓
部署频率 < 每周 1 次？
  ├─ 否 → 继续使用单体
  └─ 是 → 继续
       │
       ↓
不同模块扩展需求不同？
  ├─ 否 → 继续使用单体
  └─ 是 → 继续
       │
       ↓
技术栈需要多样化？
  ├─ 否 → 考虑模块化单体
  └─ 是 → 采用微服务
```

### 关键要点

1. **不要为了微服务而微服务** - 单体架构在很多场景下仍然适用
2. **渐进式演进** - 从模块化单体开始，逐步拆分
3. **自动化优先** - CI/CD、监控、告警必须先行
4. **团队结构匹配** - Conways 定律：系统架构反映团队沟通结构
5. **接受复杂性** - 微服务不是银弹，会带来新的复杂性

### 推荐学习路径

```
Week 1-2: 理解核心概念
  - 阅读微服务设计模式
  - 学习 DDD 基础
  
Week 3-4: 技术选型
  - 评估团队技术栈
  - 搭建开发环境
  
Week 5-8: 试点项目
  - 选择 1-2 个服务试点
  - 建立 CI/CD 流程
  
Week 9-12: 逐步迁移
  - 按业务领域逐步拆分
  - 建立监控体系
  
Month 4+: 持续优化
  - 性能调优
  - 成本优化
  - 架构演进
```
