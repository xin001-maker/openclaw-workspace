### 2. 容量估算 (Estimation)

```python
# 常用估算公式

# 存储容量
存储总量 = 日增量 × 天数 × 冗余系数

# QPS 估算
写 QPS = 日写请求数 / 86400
读 QPS = 日读请求数 / 86400
峰值 QPS = 平均 QPS × 峰值系数 (通常 3-10)

# 带宽估算
带宽 = QPS × 单次请求大小

# 服务器数量
服务器数 = 总 QPS / 单服务器 QPS 能力
```

```
示例：Twitter 容量估算

已知条件：
- DAU: 2 亿
- 日活发帖率：10%
- 平均每条推文：1KB
- 读写比：10:1

计算：
1. 日发帖量 = 2 亿 × 10% = 2000 万条/天
2. 写 QPS = 2000 万 / 86400 ≈ 230 QPS
3. 峰值写 QPS = 230 × 5 ≈ 1200 QPS
4. 读 QPS = 230 × 10 = 2300 QPS
5. 峰值读 QPS = 2300 × 5 ≈ 12000 QPS

存储：
- 日增存储 = 2000 万 × 1KB = 20GB/天
- 年存储 = 20GB × 365 = 7.3TB/年
- 5 年存储 = 7.3TB × 5 × 3 (冗余) ≈ 110TB

结论：
- 写压力不大，单集群可承受
- 读压力大，需要缓存 + CDN
- 存储可控，普通数据库即可
```

### 2. 缓存策略

```python
# 缓存更新策略对比

┌─────────────────────────────────────────────────────┐
│  策略          │ 描述 │ 优点 │ 缺点 │ 适用场景     │
├─────────────────────────────────────────────────────┤
│  Cache Aside  │ 先 DB 后删缓存 │ 简单 │ 短暂不一致 │ 通用  │
│  Read Through │ 缓存负责加载 │ 一致性好 │ 复杂 │ 读多写少 │
│  Write Through│ 缓存同步写 DB │ 强一致 │ 写延迟高 │ 金融  │
│  Write Behind │ 缓存异步写 DB │ 写性能好 │ 可能丢失 │ 高并发│
└─────────────────────────────────────────────────────┘

# 缓存问题与解决

1. 缓存穿透 (查询不存在的数据)
   解决：布隆过滤器、缓存空值

2. 缓存击穿 (热点 key 过期)
   解决：互斥锁、永不过期

3. 缓存雪崩 (大量 key 同时过期)
   解决：随机过期时间、多级缓存
```

### 题目 1：设计 Twitter

```
┌─────────────────────────────────────────────────────┐
│              Twitter 系统设计                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  核心挑战：                                          │
│  1. Feed 流如何高效生成？                           │
│  2. 大 V 发推如何不拖垮系统？                       │
│  3. 如何支持实时性？                                │
│                                                     │
│  Feed 流生成方案对比：                               │
│                                                     │
│  方案 1: 推模式 (Push)                               │
│  - 发推时写入所有粉丝的 Feed                         │
│  - 优点：读快                                        │
│  - 缺点：大 V 发推慢、写扩散                         │
│  - 适用：粉丝数少的场景                              │
│                                                     │
│  方案 2: 拉模式 (Pull)                               │
│  - 发推只写自己的时间线                              │
│  - 读时拉取关注用户的推文合并                        │
│  - 优点：发推快、写少                                │
│  - 缺点：读慢、需要合并排序                          │
│  - 适用：粉丝数多的场景                              │
│                                                     │
│  方案 3: 混合模式 (Hybrid) ⭐推荐                    │
│  - 普通用户：推模式                                  │
│  - 大 V (>5000 粉丝): 拉模式                          │
│  - 结合两者优点                                      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

```python
# Twitter 核心代码设计

class TwitterService:
    def __init__(self):
        self.db = get_db()
        self.redis = get_redis()
        self.mq = get_kafka()
        self.THRESHOLD = 5000  # 大 V 阈值
    
    def post_tweet(self, user_id, content):
        # 1. 写入推文表
        tweet = self.db.tweets.insert({
            'user_id': user_id,
            'content': content,
            'created_at': time.time()
        })
        
        # 2. 获取粉丝数
        follower_count = self.db.follows.count_followers(user_id)
        
        # 3. 根据粉丝数选择模式
        if follower_count < self.THRESHOLD:
            # 推模式：写入粉丝 Feed
            self._push_to_followers(user_id, tweet)
        else:
            # 拉模式：只写事件
            self.mq.send('tweet_created', {
                'user_id': user_id,
                'tweet_id': tweet.id
            })
        
        return tweet
    
    def get_feed(self, user_id, limit=20):
        # 1. 先查缓存
        feed = self.redis.zrange(f'feed:{user_id}', 0, limit-1)
        if feed:
            return feed
        
        # 2. 缓存未命中，查 DB
        # 获取关注列表
        following = self.db.follows.get_following(user_id)
        
        # 拉取推文
        tweets = []
        for follow_id in following:
            user_tweets = self.db.tweets.get_user_tweets(
                follow_id, limit=limit
            )
            tweets.extend(user_tweets)
        
        # 按时间排序
        tweets.sort(key=lambda t: t.created_at, reverse=True)
        tweets = tweets[:limit]
        
        # 3. 写入缓存
        for i, tweet in enumerate(tweets):
            self.redis.zadd(f'feed:{user_id}', tweet.id, tweet.created_at)
        
        return tweets
```

```python
# 短链服务实现

class ShortUrlService:
    BASE62 = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"
    
    def __init__(self):
        self.redis = get_redis()
        self.db = get_db()
        self.snowflake = Snowflake()
    
    def shorten(self, long_url: str) -> str:
        # 1. 检查是否已存在
        short_code = self.redis.get(f"url:long:{long_url}")
        if short_code:
            return short_code
        
        # 2. 生成唯一 ID
        unique_id = self.snowflake.next_id()
        
        # 3. Base62 编码
        short_code = self._encode_base62(unique_id)
        
        # 4. 存储映射 (缓存 + DB)
        self.redis.setex(f"url:short:{short_code}", long_url, 30*86400)
        self.redis.setex(f"url:long:{long_url}", short_code, 30*86400)
        
        self.db.urls.insert({
            'short_code': short_code,
            'long_url': long_url,
            'created_at': time.time()
        })
        
        return short_code
    
    def redirect(self, short_code: str) -> str:
        # 1. 查缓存
        long_url = self.redis.get(f"url:short:{short_code}")
        
        if not long_url:
            # 2. 查 DB
            record = self.db.urls.get_by_short_code(short_code)
            if record:
                long_url = record.long_url
                # 回写缓存
                self.redis.setex(f"url:short:{short_code}", long_url, 30*86400)
        
        if long_url:
            # 3. 异步记录访问
            self._record_access(short_code)
        
        return long_url
    
    def _encode_base62(self, num: int) -> str:
        if num == 0:
            return self.BASE62[0]
        
        result = []
        while num > 0:
            result.append(self.BASE62[num % 62])
            num //= 62
        
        return ''.join(reversed(result))
```

### 面试模拟对话

```
面试官：我们来设计一个 Instagram，你觉得核心功能是什么？

候选人：好的。Instagram 的核心功能我认为是：
1. 上传和浏览照片/视频
2. Feed 流展示
3. 点赞和评论

为了简化，我们聚焦在前两个功能。请问 DAU 大概多少？

面试官：假设 1 亿 DAU。

候选人：好的。让我先做一下容量估算：
- 假设 50% 用户每天发帖，就是 5000 万条/天
- 平均每条 500KB (照片压缩后)
- 日增存储：5000 万 × 500KB ≈ 25TB/天
- 读写比假设 100:1 (浏览远多于发帖)

基于这个规模，我建议：
1. 文件存储用 S3 + CDN
2. 元数据用 MySQL 分库分表
3. Feed 流用 Redis + 推模式

您觉得这个方向对吗？

面试官：方向可以。详细说说 Feed 流怎么设计？

候选人：Feed 流我建议使用推模式：
1. 用户发帖后，异步写入所有粉丝的 Feed 列表
2. Feed 列表用 Redis Sorted Set 存储
3. 读取时直接从 Redis 按时间排序拉取

对于大 V (比如粉丝>1 万)，可以改用拉模式或混合模式，
避免写扩散问题。

面试官：如果 Redis 挂了怎么办？

候选人：好问题。我会这样保证高可用：
1. Redis Cluster 多主多从
2. 本地缓存兜底 (Caffeine)
3. 降级策略：Redis 不可用时，直接查 DB

同时我们会监控 Redis 的各项指标，提前预警。
```

## 常见错误

### ❌ 错误 1：直接跳进细节

```
❌ 错误示范
"我们用 Kafka，因为它性能好..."
"数据库用 MySQL，分 8 个库..."

✅ 正确做法
先问需求 → 估算容量 → 高层设计 → 再深入细节
```

### ❌ 错误 2：忽视权衡分析

```
❌ 错误示范
"我们就用 Redis 缓存"

✅ 正确做法
"可以用 Redis 或 Memcached：
- Redis 支持更多数据结构，但更复杂
- Memcached 更简单，但功能少
考虑到我们需要 Sorted Set 做 Feed 流，建议用 Redis"
```

### ❌ 错误 3：不考虑扩展性

```
❌ 错误示范
设计时只考虑当前规模

✅ 正确做法
"当前规模单库可以承受，但考虑到增长：
- 预留分库分表方案
- 使用一致性 Hash 便于扩容
- 监控指标达到阈值时自动告警"
```

### ❌ 错误 4：忽视故障处理

```
❌ 错误示范
只设计正常流程

✅ 正确做法
"正常流程是...
异常处理：
- 服务超时：重试 + 降级
- 数据库挂了：主从切换
- 缓存穿透：布隆过滤器"
```

## 总结

### 解题模板

```
1. 需求澄清 (5 分钟)
   - 用户规模、核心功能、性能要求

2. 容量估算 (5 分钟)
   - QPS、存储、带宽

3. 高层设计 (10 分钟)
   - 核心组件、数据流向

4. 详细设计 (15 分钟)
   - API、数据模型、关键算法

5. 扩展讨论 (10 分钟)
   - 性能优化、容错、监控
```

### 核心原则

- ✅ 先问需求，不要假设
- ✅ 自顶向下，先整体后细节
- ✅ 主动分析 trade-off
- ✅ 考虑扩展性和容错
- ✅ 保持沟通，不要闷头设计
