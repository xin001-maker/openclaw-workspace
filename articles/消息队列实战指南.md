## 为什么需要消息队列

### 场景一：异步处理

```
┌─────────────────────────────────────────────────────┐
│              同步流程 (无 MQ)                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  用户注册 → 写入数据库 → 发送邮件 → 发送短信 → 返回  │
│     │         (100ms)    (500ms)     (300ms)        │
│     └─────────────────────────────────────────┘     │
│                    总耗时：900ms                     │
│                                                     │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│              异步流程 (使用 MQ)                      │
├─────────────────────────────────────────────────────┤
│                                                     │
│  用户注册 → 写入数据库 → 发送消息 → 立即返回        │
│     │         (100ms)    (10ms)      (110ms)        │
│     │                                              │
│     └─→ 消息队列                                    │
│              ↓              ↓                       │
│         邮件服务      短信服务                       │
│         (异步消费)    (异步消费)                     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**收益：**
- ✅ 响应时间从 900ms 降至 110ms (87% ↓)
- ✅ 核心业务与辅助业务解耦
- ✅ 邮件/短信服务故障不影响注册

### 场景二：流量削峰

```
┌─────────────────────────────────────────────────────┐
│              秒杀场景                                │
├─────────────────────────────────────────────────────┤
│                                                     │
│  请求量                                             │
│    ↑                                                │
│    │    ╱╲                                          │
│    │   ╱  ╲     ╱╲                                  │
│    │  ╱    ╲   ╱  ╲                                 │
│    │ ╱      ╲ ╱    ╲                                │
│    │╱        ╱      ╲                               │
│    └────────────────────→ 时间                      │
│                                                     │
│  数据库承受能力：───────                             │
│                                                     │
│  问题：峰值请求 > 数据库处理能力 → 系统崩溃          │
│                                                     │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│              使用 MQ 削峰                            │
├─────────────────────────────────────────────────────┤
│                                                     │
│  请求 → 消息队列 (缓冲) → 数据库 (匀速消费)         │
│                                                     │
│  收益：                                              │
│  - 保护数据库不被突发流量击垮                        │
│  - 按系统最大能力匀速处理                            │
│  - 失败请求可重试                                    │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 场景三：系统解耦

```
┌─────────────────────────────────────────────────────┐
│              紧耦合架构 (无 MQ)                      │
├─────────────────────────────────────────────────────┤
│                                                     │
│  订单服务 ──→ 库存服务                               │
│      │                                              │
│      ├────→ 支付服务                                 │
│      │                                              │
│      ├────→ 物流服务                                 │
│      │                                              │
│      └────→ 通知服务                                 │
│                                                     │
│  问题：                                              │
│  - 订单服务需要知道所有下游服务                      │
│  - 新增服务需要修改订单服务                          │
│  - 某个服务故障影响整个链路                          │
│                                                     │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│              解耦架构 (使用 MQ)                      │
├─────────────────────────────────────────────────────┤
│                                                     │
│  订单服务 ──→ 订单创建事件                           │
│                  │                                  │
│      ┌───────────┼───────────┬──────────┐          │
│      ↓           ↓           ↓          ↓          │
│  库存服务    支付服务    物流服务    通知服务        │
│  (订阅)      (订阅)      (订阅)      (订阅)         │
│                                                     │
│  收益：                                              │
│  - 订单服务只负责发布事件                            │
│  - 新增服务只需订阅事件                              │
│  - 服务间完全解耦                                    │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 核心概念解析

### 基本术语

```
┌─────────────────────────────────────────────────────┐
│              消息队列核心概念                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  Producer (生产者)                                  │
│  └─ 发送消息的应用                                  │
│                                                     │
│  Consumer (消费者)                                  │
│  └─ 接收并处理消息的应用                            │
│                                                     │
│  Queue (队列)                                       │
│  └─ 消息的容器，FIFO 顺序                           │
│                                                     │
│  Topic (主题)                                       │
│  └─ 消息的分类/频道                                 │
│                                                     │
│  Exchange (交换机) - RabbitMQ 特有                  │
│  └─ 接收生产者消息，路由到队列                      │
│                                                     │
│  Partition (分区) - Kafka 特有                      │
│  └─ Topic 的物理分片，支持并行处理                  │
│                                                     │
│  Consumer Group (消费组)                            │
│  └─ 一组消费者，共同消费一个 Topic                  │
│                                                     │
│  Offset (偏移量)                                    │
│  └─ 消息在分区中的位置标识                          │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 消息模式

#### 1. 点对点 (Point-to-Point)

```
Producer → Queue → Consumer
              │
              └─ 消息只能被一个消费者消费
```

**特点：**
- 消息消费后从队列删除
- 一个消息只能被一个消费者处理
- 适合任务分发场景

#### 2. 发布订阅 (Pub/Sub)

```
Producer → Topic → Consumer1
                   Consumer2
                   Consumer3
                   └─ 所有订阅者都收到消息
```

**特点：**
- 消息可被多个消费者消费
- 支持广播模式
- 适合事件通知场景

#### 3. 竞争消费者 (Competing Consumers)

```
              → Consumer1
Producer → Queue → Consumer2
              → Consumer3
              └─ 多个消费者竞争消费
```

**特点：**
- 提高消费吞吐量
- 自动负载均衡
- 适合高并发处理

| 特性 | RabbitMQ | Kafka | Redis Stream | RocketMQ |
|------|----------|-------|--------------|----------|

| 语言 | Erlang | Scala/Java | C | Java |

| 可靠性 | 高 | 高 | 中 | 高 |

| 社区活跃度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 学习曲线 | 中 | 陡 | 低 | 中 |
| 适用场景 | 企业应用 | 大数据流 | 简单场景 | 互联网 |

### 选型建议

```
选择 RabbitMQ 如果：
✅ 需要复杂的路由规则
✅ 对延迟敏感 (微秒级)
✅ 企业级应用，需要管理界面
✅ 团队熟悉 Erlang 系技术

选择 Kafka 如果：
✅ 需要处理海量数据 (百万级 QPS)
✅ 日志收集、流处理场景
✅ 需要消息持久化和回溯
✅ 大数据生态集成

选择 Redis Stream 如果：
✅ 已有 Redis，不想引入新组件
✅ 简单场景，低延迟要求
✅ 消息量不大 (万级 QPS)
✅ 快速原型开发

选择 RocketMQ 如果：
✅ 阿里技术栈
✅ 需要金融级可靠性
✅ 大规模分布式场景
✅ 需要事务消息
```

### 工作队列 (Work Queue)

```python
# producer.py - 发送任务
import pika
import json

# 连接 RabbitMQ
connection = pika.BlockingConnection(
    pika.ConnectionParameters('localhost')
)
channel = connection.channel()

# 声明队列 (持久化)
channel.queue_declare(queue='task_queue', durable=True)

# 发送消息
for i in range(10):
    task = {'id': i, 'content': f'Task {i}'}
    channel.basic_publish(
        exchange='',
        routing_key='task_queue',
        body=json.dumps(task),
        properties=pika.BasicProperties(
            delivery_mode=2,  # 消息持久化
        )
    )
    print(f"Sent: {task}")

connection.close()

# consumer.py - 消费任务
import pika
import json
import time

def callback(ch, method, properties, body):
    task = json.loads(body)
    print(f"Received: {task}")
    
    # 模拟处理
    time.sleep(1)
    
    print(f"Completed: {task}")
    # 手动确认
    ch.basic_ack(delivery_tag=method.delivery_tag)

connection = pika.BlockingConnection(
    pika.ConnectionParameters('localhost')
)
channel = connection.channel()

channel.queue_declare(queue='task_queue', durable=True)

# 公平分发 (一次只处理一个消息)
channel.basic_qos(prefetch_count=1)

channel.basic_consume(
    queue='task_queue',
    on_message_callback=callback
)

print("Waiting for messages...")
channel.start_consuming()
```

### 发布订阅 (Pub/Sub)

```python
# 扇形交换机 - 广播模式
# producer.py
import pika

connection = pika.BlockingConnection(
    pika.ConnectionParameters('localhost')
)
channel = connection.channel()

# 声明扇形交换机
channel.exchange_declare(
    exchange='logs',
    exchange_type='fanout'
)

# 发送消息 (不指定 routing_key)
message = "System alert: High CPU usage!"
channel.basic_publish(
    exchange='logs',
    routing_key='',
    body=message
)
print(f"Sent: {message}")

connection.close()

# consumer.py - 邮件服务
connection = pika.BlockingConnection(
    pika.ConnectionParameters('localhost')
)
channel = connection.channel()

channel.exchange_declare(
    exchange='logs',
    exchange_type='fanout'
)

# 创建临时队列 (随机名称)
result = channel.queue_declare(queue='', exclusive=True)
queue_name = result.method.queue

# 绑定交换机
channel.queue_bind(
    exchange='logs',
    queue=queue_name
)

def callback(ch, method, properties, body):
    print(f"Email service received: {body}")
    # 发送邮件逻辑
    send_email(body)

channel.basic_consume(
    queue=queue_name,
    on_message_callback=callback,
    auto_ack=True
)

channel.start_consuming()
```

### 路由模式 (Routing)

```python
# 直连交换机 - 根据 routing_key 路由
# producer.py
import pika
import sys

connection = pika.BlockingConnection(
    pika.ConnectionParameters('localhost')
)
channel = connection.channel()

channel.exchange_declare(
    exchange='direct_logs',
    exchange_type='direct'
)

# 根据日志级别路由
severity = sys.argv[1] if len(sys.argv) > 1 else 'info'
message = ' '.join(sys.argv[2:]) or 'Log message'

channel.basic_publish(
    exchange='direct_logs',
    routing_key=severity,
    body=message
)
print(f"Sent {severity}: {message}")

connection.close()

# consumer.py - 只接收 error 和 warning
connection = pika.BlockingConnection(
    pika.ConnectionParameters('localhost')
)
channel = connection.channel()

channel.exchange_declare(
    exchange='direct_logs',
    exchange_type='direct'
)

result = channel.queue_declare(queue='', exclusive=True)
queue_name = result.method.queue

# 绑定多个 routing_key
for severity in ['error', 'warning']:
    channel.queue_bind(
        exchange='direct_logs',
        queue=queue_name,
        routing_key=severity
    )

def callback(ch, method, properties, body):
    print(f"Received: {body}")

channel.basic_consume(
    queue=queue_name,
    on_message_callback=callback,
    auto_ack=True
)

channel.start_consuming()
```

### 主题模式 (Topic)

```python
# 主题交换机 - 支持通配符路由
# routing_key 格式：app.component.level
# 通配符：* (一个词), # (零个或多个词)

# producer.py
import pika

connection = pika.BlockingConnection(
    pika.ConnectionParameters('localhost')
)
channel = connection.channel()

channel.exchange_declare(
    exchange='topic_logs',
    exchange_type='topic'
)

# 发送不同路由键的消息
messages = [
    ('user.created.info', 'User 123 created'),
    ('user.deleted.error', 'Failed to delete user'),
    ('order.created.info', 'Order 456 created'),
    ('order.payment.warning', 'Payment delayed'),
]

for routing_key, body in messages:
    channel.basic_publish(
        exchange='topic_logs',
        routing_key=routing_key,
        body=body
    )
    print(f"Sent {routing_key}: {body}")

connection.close()

# consumer.py - 订阅所有 user 相关消息
connection = pika.BlockingConnection(
    pika.ConnectionParameters('localhost')
)
channel = connection.channel()

channel.exchange_declare(
    exchange='topic_logs',
    exchange_type='topic'
)

result = channel.queue_declare(queue='', exclusive=True)
queue_name = result.method.queue

# 绑定：user.*.* (所有 user 相关)
channel.queue_bind(
    exchange='topic_logs',
    queue=queue_name,
    routing_key='user.*'
)

# 绑定：*.error (所有 error 级别)
channel.queue_bind(
    exchange='topic_logs',
    queue=queue_name,
    routing_key='*.error'
)

channel.start_consuming()
```

### Spring Boot 集成

```java
// application.yml
spring:
  rabbitmq:
    host: localhost
    port: 5672
    username: admin
    password: admin123
    listener:
      simple:
        acknowledge-mode: manual  # 手动确认
        prefetch: 1  # 公平分发

// 配置类
@Configuration
public class RabbitConfig {
    
    @Bean
    public Queue taskQueue() {
        return new Queue("task_queue", true);
    }
    
    @Bean
    public TopicExchange topicExchange() {
        return new TopicExchange("topic_logs");
    }
    
    @Bean
    public Binding binding(Queue taskQueue, TopicExchange topicExchange) {
        return BindingBuilder.bind(taskQueue)
            .to(topicExchange)
            .with("user.*");
    }
}

// 生产者
@Service
public class MessageProducer {
    
    @Autowired
    private RabbitTemplate rabbitTemplate;
    
    public void sendUserEvent(String eventType, User user) {
        String routingKey = "user." + eventType + ".info";
        rabbitTemplate.convertAndSend(
            "topic_logs",
            routingKey,
            user
        );
    }
}

// 消费者
@Component
public class MessageConsumer {
    
    @RabbitListener(queues = "task_queue")
    public void handleTask(Task task, Channel channel, 
                          @Header(AmqpHeaders.DELIVERY_TAG) long tag) 
            throws Exception {
        try {
            // 处理任务
            processTask(task);
            // 手动确认
            channel.basicAck(tag, false);
        } catch (Exception e) {
            // 拒绝消息，重新入队
            channel.basicNack(tag, false, true);
        }
    }
}
```

### 生产者示例

```python
from kafka import KafkaProducer
import json
from datetime import datetime

# 创建生产者
producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8'),
    key_serializer=lambda k: k.encode('utf-8') if k else None,
    # acks 配置
    acks='all',  # 所有副本确认
    retries=3,   # 重试次数
    compression_type='gzip',  # 压缩
)

# 发送消息
def send_order_event(order_id, order_data):
    future = producer.send(
        'orders',  # Topic
        key=str(order_id),
        value={
            'order_id': order_id,
            'user_id': order_data['user_id'],
            'amount': order_data['amount'],
            'timestamp': datetime.now().isoformat()
        }
    )
    
    # 异步回调
    future.add_callback(
        lambda metadata: print(
            f"Sent to partition {metadata.partition}, "
            f"offset {metadata.offset}"
        )
    )
    future.add_errback(
        lambda exception: print(f"Send failed: {exception}")
    )

# 批量发送
for i in range(100):
    send_order_event(i, {'user_id': 1, 'amount': 100})

# 刷新并关闭
producer.flush()
producer.close()
```

### 消费者示例

```python
from kafka import KafkaConsumer
import json

# 创建消费者
consumer = KafkaConsumer(
    'orders',  # Topic
    bootstrap_servers=['localhost:9092'],
    auto_offset_reset='earliest',  # 从最早开始消费
    enable_auto_commit=False,  # 手动提交
    group_id='order-processor',  # 消费组
    value_deserializer=lambda v: json.loads(v.decode('utf-8')),
    key_deserializer=lambda k: k.decode('utf-8') if k else None,
)

# 订阅多个 Topic
consumer.subscribe(['orders', 'payments'])

# 消费消息
for message in consumer:
    print(
        f"Received: key={message.key}, "
        f"value={message.value}, "
        f"partition={message.partition}, "
        f"offset={message.offset}"
    )
    
    # 处理业务逻辑
    process_order(message.value)
    
    # 手动提交 offset
    consumer.commit()

# 手动指定分区
from kafka import TopicPartition

consumer.assign([
    TopicPartition('orders', 0),
    TopicPartition('orders', 1),
])
```

### Spring Boot 集成

```java
// application.yml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: order-processor
      auto-offset-reset: earliest
      enable-auto-commit: false
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.JsonDeserializer
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.JsonSerializer
      acks: all
      retries: 3

// 生产者
@Service
public class OrderProducer {
    
    @Autowired
    private KafkaTemplate<String, Order> kafkaTemplate;
    
    public void sendOrder(Order order) {
        kafkaTemplate.send(
            "orders",
            String.valueOf(order.getId()),
            order
        ).addCallback(
            success -> log.info("Sent: {}", success.getRecordMetadata()),
            failure -> log.error("Failed: {}", failure.getMessage())
        );
    }
}

// 消费者
@Component
public class OrderConsumer {
    
    @KafkaListener(
        topics = "orders",
        groupId = "order-processor"
    )
    public void consume(
        @Payload Order order,
        @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
        @Header(KafkaHeaders.OFFSET) long offset,
        Acknowledgment ack
    ) {
        log.info(
            "Received: order={}, partition={}, offset={}",
            order, partition, offset
        );
        
        try {
            processOrder(order);
            // 手动确认
            ack.acknowledge();
        } catch (Exception e) {
            log.error("Processing failed", e);
            // 可配置重试策略
            throw e;
        }
    }
}
```

### 消费者组

```python
import redis
import time
from datetime import datetime

class StreamConsumer:
    def __init__(self, stream, group, consumer):
        self.r = redis.Redis()
        self.stream = stream
        self.group = group
        self.consumer = consumer
        
        # 创建消费组 (如果不存在)
        try:
            self.r.xgroup_create(
                stream, group, id='$', mkstream=True
            )
        except redis.exceptions.ResponseError:
            pass  # 组已存在
    
    def consume(self):
        while True:
            messages = self.r.xreadgroup(
                groupname=self.group,
                consumername=self.consumer,
                streams={self.stream: '>'},
                count=10,
                block=5000
            )
            
            if not messages:
                # 检查待处理消息 (失败重试)
                self.handle_pending()
                continue
            
            for stream, msgs in messages:
                for msg_id, fields in msgs:
                    try:
                        self.process_message(fields)
                        # 确认
                        self.r.xack(stream, self.group, msg_id)
                    except Exception as e:
                        print(f"Processing failed: {e}")
                        # 不确认，稍后重试
    
    def handle_pending(self):
        # 处理超过 1 分钟未确认的消息
        pending = self.r.xpending_range(
            self.stream,
            self.group,
            min='-',
            max='+',
            count=10
        )
        
        for p in pending:
            if p['time_since_delivered'] > 60000:  # 1 分钟
                # 重新分配给自己处理
                msgs = self.r.xclaim(
                    self.stream,
                    self.group,
                    self.consumer,
                    min_idle_time=60000,
                    message_ids=[p['message_id']]
                )
                # 处理重新分配的消息
```

## 消息可靠性保证

### 1. 消息不丢失

```
┌─────────────────────────────────────────────────────┐
│              消息丢失的三个环节                      │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. 生产者 → MQ (发送失败)                          │
│     解决：确认机制 + 重试                            │
│                                                     │
│  2. MQ 内部 (存储失败)                               │
│     解决：持久化 + 副本                              │
│                                                     │
│  3. MQ → 消费者 (消费失败)                          │
│     解决：手动 ACK + 重试队列                        │
│                                                     │
└─────────────────────────────────────────────────────┘
```

#### RabbitMQ 可靠性配置

```python
# 生产者确认
channel.confirm_delivery()

    channel.basic_publish(

            requeue=True
        )

    queue='task_queue',
    on_message_callback=callback
)
```

### 3. 死信队列

```python
# RabbitMQ 死信队列配置
channel.queue_declare(
    queue='dead_letter_queue',
    durable=True
)

channel.queue_declare(
    queue='task_queue',
    durable=True,
    arguments={
        'x-dead-letter-exchange': '',
        'x-dead-letter-routing-key': 'dead_letter_queue',
        'x-message-ttl': 60000,  # 1 分钟超时
    }
)

# 消费者重试逻辑
def callback(ch, method, properties, body):
    retry_count = properties.headers.get('x-retry-count', 0)
    
    try:
        process(body)
        ch.basic_ack(delivery_tag=method.delivery_tag)
    except Exception as e:
        if retry_count < 3:
            # 重新入队，增加重试计数
            properties.headers['x-retry-count'] = retry_count + 1
            ch.basic_publish(
                exchange='',
                routing_key='task_queue',
                body=body,
                properties=properties
            )
            ch.basic_ack(delivery_tag=method.delivery_tag)
        else:
            # 超过最大重试次数，进入死信队列
            ch.basic_reject(
                delivery_tag=method.delivery_tag,
                requeue=False
            )
```
